Actually, your trace of the Empress shows that it did not "almost make it"; in fact, it, too, never succeeded in sending a large packet and having it acknowledged.  The traffic in the trace is entirely (a) routing packets from the gateway, (b) successful printer status interchanges (requires small packets only), and (c) many retransmissions of the first page of the press file you were trying to send.'
'
You say "Leaf does work on her Dolphin by virtue of being able to COPYFILE".  Insufficient evidence.  Can she WRITE files using Leaf?  Reading files using Leaf does not require transmitting large packets.  E.g.,  set \FTPAVAILABLE to NIL and then try the COPYFILE to your server again.'
'
Some more tests to try to see who is at fault in not transmitting large packets':'
'
Load ETHERRECORDS.'
(ADVISE ''(PUTPUPSTRING IN PUP.ECHOUSER) ''AFTER ''(replace PUPLENGTH of PUP with 200))'
'
Then try calling (PUP.ECHOUSER host) to echo to (a) another machine (e.g. a DLion running Lisp) on the same net, (b) the local gateway, (c) your IFS on another net.  If this works (or doesn''t), try adjusting the constant 200 in the advice anywhere from 50 to 532 (be sure to unadvise it first before advising it again).'
'
You can do similar tests with NS echoing':'
'
(ADVISE ''(XIPAPPEND.STRING IN NS.ECHOUSER) ''AFTER ''(replace XIPLENGTH of EPKT with 200))'
'
and then call (NS.ECHOUSER "net#a.b.c#") for suitable NS addresses.  Adjust the constant 200 similarly.'
